## Bug Reporting Guidelines

### When Reporting a Bug

1. **Title**: Clear, concise description
2. **Severity**: Critical, High, Medium, or Low
3. **Area**: Frontend, Backend, Integration, etc.
4. **Steps to Reproduce**: Step-by-step instructions
5. **Expected Behavior**: What should happen
6. **Actual Behavior**: What actually happens
7. **Environment**: Browser, OS, version
8. **Screenshots/Logs**: When applicable

### Bug Triage Process

1. Bug reported in `docs/findings.md`
2. Architect Agent triages and categorizes
3. Bug added to `docs/bugs.md`
4. GitHub Issue created with labels
5. Assigned to appropriate specialist
6. Specialist fixes and closes issue
7. Bug removed from `docs/bugs.md`

## Bug Prevention

### Best Practices

- **Code Review**: All changes must be reviewed
- **Automated Testing**: Unit and integration tests
- **Type Safety**: Strict TypeScript, no `any`
- **Linting**: ESLint and Prettier
- **Documentation**: Clear comments and JSDoc

### Quality Gates

- All tests passing
- No TypeScript errors
- No linting errors
- Code review approved
- Performance metrics met
